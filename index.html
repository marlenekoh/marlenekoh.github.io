VIMU

Team members: Deborah Low, Koh Yee Ru, Marlene Koh, Ng Xin Ler
GitHub | Lo-Fi Prototype | Hi-Fi Prototype

This project was done as a part of the module CS3240: Interation Design under the National University of Singapore (NUS) in 2018. Students are tasked to come up with a UX solution to a current problem.

Problem and scope

Controllers have been a popular way for DJs and musicians to create their own music with digital sounds. Controllers such as the Midi Fighter, Launchpad, Ableton Push have risen in demand along with the rising need for reliable instruments on stage and studio. However, not only can they be expensive, they are also required to install a software to set up, which complicates the mixing process. The hassle of having to bring along the controller and their laptop is another pain point for the users.
Currently, there are web apps like Mikutap and Patatap that allows creation of music. However, they do not support mapping of local audio, as well as recording/downloading the song composed. This hence limits the creativity of musicians as they are now restricted by the default sounds available to them.


Vision and Target Audience

Vimu aims to provide a platform for new musicians to familiarize themselves with the core features of controllerism software while keeping the aesthetic appeal controllers provides for onlookers.


User Study and Research

We decided to scope our product to the club members of the National University of Singapore (NUS) Electronic Music Lab (EML). EML is a music performance group that creates, performs and promotes original electronic music. Club members have the opportunity to work with a music mixing software called Ableton. We interviewed 3 new club members and 3 seniors to a total of 6.
My team member Xin Ler created the following two personas to summarise the characteristics of our target audience.

    Key pain points faced:
  • Steep learning curve for new club members to pick up the software due to the immense number of features
  • Lack of portability, as they need to carry around a controller in addition to their laptops
  • Long time taken to produce a soundtrack due to the need of switching between screens
  • Lack of visuals during performances as the controllerâ€™s lights cannot be projected onto the screen
  • Difficult to tell if a key has an existing sound file mapped to it

Solution

We decided to create a web application that allows users to simulate the look and feel of controllers and their software through their laptop keyboards and screen displays.

    Core features:
  • Existing sound library
  • Upload custom sound clips
  • Map sound clips to keyboard keys
  • Map visuals to keyboard keys
  • Basic sound editing features
  • Customise visuals
  • Performance mode
  • Customise on-screen keyboard
  • Portability - our web application is accessible on any device with internet connection

Based on the core features we have come up with, the four of us split up to brainstorm and ideate for an initial prototype.
I grouped the features into 3 separate tabs: Upload, Map, Mix (Edit) and dedicated tab to focus on one task.

Upload was to be implemented with a drag and drop/click to upload functionality. The Map tab was further split into 2 tabs: Audio and Visual, to cater to the requirement of letting users customise what sound/audio was to be mapped to which key. The Mix tab was purely to edit audio.
My sound library took up majority of the screen space in the Map tab. Sound files were displayed as large cards with the sound name and sound length.

The sound and visual key mapping done in my initial prototype was a click and click function. I also added a collapsible keyboard which allowed the user to hide/display the keyboard when necessary in order to increase the amount of space the user has to work with when browsing the online sound library.

Evaluation of Initial Prototype

After discussion, we decided to turn VIMU into a single page application based on Yee Ru's intial idea. Many of the pages I had initially designed could have been simplified and sectioned out into a single page. Having the different Upload, Map and Mix tabs displayed as sections in the same screen improved user's visibility and navigability to the different features. Creating a single page application also resulted in less space wasted.

Another change would be that we decided to make the audio/visual key mapping process into a drag and drop function. My team member Yee Ru created the following design for the drag and drop feature:

Having the differently-coloured rectangle on the key increases affordance of the drag and drop functionality - sound files can be dragged from the sound library to the key and from keys to other keys.

I also decided to change the design of the sound library from grid format to list format to include only the necessary information (name) and reduce wastage of space.

Using Nielsen's Heuristics

H6 - Recognition rather than recall
To target one of the pain points where it is difficult for users to verify if a key has an existing sound file mapped to it, I displayed the name of the sound file on the key. This allows users to quickly distinguish between keys without any sounds and keys with sounds.

When the user edits a key, the key will show up in white on the on-screen keyboard, and will also be displayed in the top right corner of the Sound Editor section. This design allows the user to recognise the current key that he/she is selecting rather than recall it.

H8 - Aesthetic and minimalist design
In comparison to my initial prototype that had too many pages and too many buttons, I designed my hi-fidelity prototype to be cleaner and more minimilistic to declutter the space. The start page now only displays the keyboard, and progressive levels of detail are provided only upon switching to Edit mode (the pink lock). Even in Edit mode, the Sound Editor and Visuals Editor remain empty unless a key is selected, which reduces the user's cognitive load.

H10 - Help and Documentation
The help icon is always readily available at the top right corner of the screen.

On clicking it, tooltips will appear with information on the function of each element.

This project was done as a part of CP2106: Independent Software Development Project under the National University of Singapore (NUS) in 2017. Students are tasked to create any project (Mobile game, Mobile app, Web app, etc) over the course of 3 months.

About Hangry

Hangry is a 2D platformer idle game featuring gesture recognition. Players take on the role of a hungry fighter, who defeats enemy monsters to earn gold. Enemy monsters can be defeated by drawing gestures on the screen in battle. Gold earned can be used to boost player stats to defeat increasingly stronger monsters as the game progresses.


Motivation

Incremental games (also known as idle games or clicker games) have been gaining popularity in recent years, since they are easy to adopt and are also designed to give high returns with little effort or input from the player. Features of incremental games include repeated actions to gain currency such as to progressively improve the character capability and increase the ease of gameplay. As a result, they are psychologically rewarding and well perceived by many people.


Screenshots


*Sprites, animation and background assets were done by me


What makes us different?

The average incremental game uses a clicking mechanism to conduct its gameplay. Although this may be an intuitive mechanism for new users to pick up, clicking will more often than not turn repetitive after some time. Using touch and drawing gestures to deal damage to opponents creates a more versatile and interactive experience for users. Our game will also include offline progress, enabling the user to gain currency without any input, a feature that most incremental games may have.

    Features:
  • Gesture recognition - $P Point-Cloud Recognizer
  • Google Play Integration
  • Save and Load System - local and cloud save
  • Offline Progression
  • Incremental Game and Setup - adaptive difficulty
  • Daily Check-in
  • Multiple Levels
    Future Plans:
  • Storyline and narrative
  • Add ally heroes to help fight battles
  • Prestige System
  • Achievements and LeaderBoard
  • Multiplayer option

Design Process

User Stories

  1. As a student with minimal time for leisure, I want to be able to play a game which allows me to destress and progress in the game at the same time, with little input from my side.
  2. As a gamer who enjoys incremental games, a mechanism other than clicking to damage opponents would make the game much more interactive.

Players can unlock more food, maps and friends as they progress through the game.

Development Process

  1. Plan and Preparation
    1. Setting the scope of what to accomplish (e.g. UI/Save-Load/Music and Sfx/Fight Scene etc.)
    2. Research on ways to create the outcome desired in 1a.
    3. Plan and draw UML diagrams to better visualise our code.
  2. Code
    1. We put into practice the knowledge we have accumulated in part 1b.
  3. Run/execute the program
    1. We tested the game on unity editor and unity remote on our phones (Android).
    2. Compile errors, bugs and problems are taken note of.
  4. Debugging and maintenance
    1. Bug fixes are prioritised based on usage and criticality (e.g. compile errors are first fixed, followed by the most frequently observed bugs).
  5. Rinse and Repeat
    1. Steps 3 and 4 are repeated until the feature is functional.
    2. Subsequently, the next cycle is started.

UX Design Process

Using Nielsen's Heuristics

H4 - Consistency and standards
Graphical User Interface (GUI) buttons are all drawn with the same colours and similar shading, to indicate that they are buttons and can be pressed. Player control buttons are located at the bottom of the screen, similar to many mobile 2D platformer games on the market.

H5 - Error Prevention
Confirmation prompts are used, for example in the event user presses the Exit Game button or wants to purchase an item.

H6 - Recognition rather than recall
The number of monsters left to defeat before the next level unlocks is always displayed at the top of the map.

H10 - Help and Documentation
Instructions on how to play the game is always readily accessible from the settings menu, which is available in all scenes except the Main Menu.

Bank Of China

Team members: Goh Rui Zhi, Marlene Koh, Ng Xin Ler, Pang Kim Jin
Prototype

This project was done as part of the NUS Case Competition 2018 UX Hackathon featuring the Bank of China. Over the span of 24 hours, students are given a list problem statement and tasked to create a prototype and user flow to target any single chosen problem.

Problem Statement

Enabling a Seamless Transaction Experience:
Currently, BOC has funds transfer functions for its own accounts transfers, 3rd party BOC accounts transfers, local bank transfers, international remittance and RMB - remittance. How can BOC build a common funds transfer screen on its Mobile Banking App to cater to all its transfer functions and deliver a simple and intuitive transactional flow?


Scenario

Jane is a BOC customer on the go, working in China. She wants to transfer money to her mother who lives in Malaysia who uses Maybank. She has the BOC app already installed on her smartphone, and opens the transfers and remittance page.

Empathy Map

Solution

Current Transaction Flow:
The current design requires user to move forward and backword between the transfers and remittance page to the select payer/payee details page. This gives the user the sensation of moving backwards in the transaction and breaks their flow of concentration.


Modified Transaction Flow:
We altered the app flow to turn it into a unidirectional flow to reduce cognitive load on users. We minimised the number of actions the user is able to take, to ensure that users to focus on one action at a time. We also added a confirmation page as a means to prevent errors.
